# Kernel 设计文档审查

## 使用场景

当用户提供一份 kernel 设计文档（GitHub issue、Markdown 文件、或文档链接），要求对其进行审查时，按本 skill 逐节检查。

---

## 第一步：获取文档

确认已拿到完整的设计文档内容。若用户只给了链接，先获取其全文。若文档不完整，在审查中标注缺失部分。

---

## 第二步：逐节评审

对以下四个必填章节依次检查。每节给出：
- **通过 / 不通过 / 部分通过**
- 具体缺失项或需改进项
- 若不通过，给出具体改进建议

---

### 章节 1：算法与代码逻辑

**检查项：**

- [ ] **核心数学公式**：是否明确写出了该 kernel 计算的数学表达式？
  - 例：注意力计算的 softmax 公式、矩阵乘的具体维度、recurrence 递推关系等
  - 判断标准：读者无需看代码即可理解 kernel 计算什么

- [ ] **实现策略**：是否说明了选择此实现方式的理由？
  - 例：为何选择在线 softmax 而非两遍扫描？为何选择分块矩阵乘而非全量？
  - 判断标准：明确写出了与备选方案的对比或取舍分析

- [ ] **关键取舍**：是否列出了主要权衡（精度 vs 速度、内存 vs 计算、通用性 vs 性能）？

**通过标准**：公式完整、选择有理由、取舍有说明。

---

### 章节 2：数据流与 Tiling 逻辑

**检查项：**

- [ ] **Tiling / 分块策略**：是否说明了分块大小及其选择依据？
  - 对 TPU：块大小是否与 VMEM 容量匹配？是否说明了如何映射到硬件内存层次？
  - 对 GPU：块大小是否考虑 SMEM 大小？是否说明了 warp 占用率？
  - 判断标准：给出了具体块大小数值及选择原因（非随意选取）

- [ ] **数据移动**：是否描述了数据如何在内存层次间流动？
  - 例：是否使用 DMA 流水线（双缓冲）？prefetch 策略？
  - 判断标准：说明了是否存在计算-通信重叠及实现方式

- [ ] **Grid 维度划分**：是否说明了并行维度的划分方式？
  - 例：哪些维度是 `parallel`，哪些是 `arbitrary`？
  - 判断标准：Grid 形状与并行策略有明确说明

- [ ] **Tensor 布局**：是否说明了关键 tensor 的内存布局（row-major / col-major / tiled）？

**通过标准**：块大小有数值且有理由，数据移动路径清晰，Grid 划分明确。

---

### 章节 3：性能目标与 Roofline 分析

**这是最严格的章节。目标：达到硬件理论峰值的 80%。**

**检查项：**

- [ ] **算术强度计算**：是否计算了该 kernel 的算术强度（FLOPs / Bytes）？
  - 公式：算术强度 = 总浮点操作数 / 总内存访问字节数
  - 判断标准：给出了具体数值，且展示了计算过程

- [ ] **瓶颈类型判断**：是否基于 roofline 模型判断了该 kernel 是 compute-bound 还是 memory-bound？
  - 判断标准：明确说明瓶颈类型，并与硬件参数对照（如 MXU 峰值、HBM 带宽）

- [ ] **80% 目标定义**：是否明确写出了具体的性能目标数值？
  - 例：目标 ≥ X TFLOPS，或目标 ≥ Y GB/s
  - 判断标准：有具体数字，不是"尽可能快"

- [ ] **本地基准测试结果**：是否包含实际测量结果？
  - 必须有：TFLOPS 或 GB/s 数值
  - 必须有：测试所用的硬件、形状、dtype 等配置
  - 判断标准：结果可复现，配置完整

- [ ] **Trace 分析**：是否包含执行时间线分析？
  - 必须有：trace viewer 截图或链接
  - 必须有：对 trace 的解读（是否有空泡、是否有重叠、瓶颈在哪）
  - 判断标准：不能只有截图而无分析

- [ ] **是否达标**：是否明确说明了当前结果是否达到 80% 目标？
  - 若未达标：是否说明了原因（硬件限制 vs 代码可优化空间）？

**通过标准**：以上 6 项全部具备。缺少基准测试数据或 trace 分析视为不通过。

**常见不通过情形：**
- 只有 roofline 分析但无实测数据
- 有数据但未说明是否达到 80%
- trace 截图无文字解读
- 配置不完整导致结果无法复现

---

### 章节 4：精度与误差分析

**检查项：**

- [ ] **误差指标**：是否给出了相对于 CPU 参考实现的误差？
  - 必须有：atol（绝对容差）、rtol（相对容差）、或最大 ULP 误差
  - 判断标准：给出了具体数值

- [ ] **梯度误差**：若 kernel 需要梯度，是否分析了反向传播的误差？
  - 判断标准：若不需要梯度，明确说明；若需要，给出梯度误差数据

- [ ] **测试方法**：是否说明了误差测试的方法论？
  - 必须有：输入分布（随机？边界值？特殊值如 inf/nan？）
  - 必须有：形状配置（测试了哪些 shape？是否覆盖边界？）
  - 判断标准：测试方法有足够覆盖度，不是单一 shape 的单次测试

- [ ] **TPU vs GPU 误差约束**：是否包含 TPU 误差 ≤ GPU 误差的验证？
  - 这是硬性约束，必须有对应的测试或分析
  - 判断标准：明确给出 TPU 误差和 GPU 误差的对比数值

**通过标准**：误差指标完整，测试方法覆盖充分，TPU ≤ GPU 约束有验证。

---

## 第三步：结构完整性检查

除四个必填章节外，检查文档是否包含：

- [ ] **Overview**：一句话说明 kernel 计算什么
- [ ] **Motivation**：为何需要这个 kernel
- [ ] **Implementation Plan**：实现步骤（理想情况下包含 7 步流程：CPU 参考 → 评审 → kernel → 测试 → API → 文档 → 基准）
- [ ] **References**：参考文献或相关 PR/issue

## 评审原则

1. **严格对待章节 3**：性能数据（基准测试 + trace）是必须项，不可用计划替代实际数据。若文档处于提案阶段（尚未实现），需明确标注哪些数据待补充，且评审结论为"条件通过"。

2. **区分提案阶段与实现阶段**：
   - 提案阶段（实现前）：算法和数据流章节可以完整，性能和精度章节可以是"计划"形式，但必须明确目标数值
   - 实现完成后提交审查：四个章节必须全部有实测数据

3. **给出可操作的改进建议**：不要只说"缺少 X"，要说"需要补充 X，具体应包含 Y 和 Z"。

4. **TPU ≤ GPU 误差约束是硬性要求**：若文档完全未提及此约束，视为不通过，必须修复。
