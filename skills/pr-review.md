# PR 代码审查

## 使用场景

当用户提供 PR 链接、diff 内容、或指定一组代码变更，要求进行代码审查时，按本 skill 逐步执行。

---

## 第一步：获取变更内容

确认已拿到完整的变更内容。可能的来源：

- 用户直接粘贴的 diff 或 patch
- PR 链接（通过当前工具可用的任意方式获取：`gh` CLI、MCP 工具、浏览器访问等）
- 当前工作区的 staged/unstaged 改动

若内容不完整（只有文件列表但无 diff 内容），先补全后再审查。

---

## 第二步：理解变更意图

在评审具体代码前，先理解这个 PR 要做什么：

- PR 标题和描述说了什么？
- 变更类型是什么（feat / fix / refactor / docs / ci / test / perf）？
- 涉及哪些模块或文件？

这一步的目的是校准评审重心——例如 fix 类型的 PR 重点看正确性，perf 类型重点看效率。

---

## 第三步：逐优先级评审

按以下顺序逐项评审，**优先级高的问题先审**。发现问题时记录文件名、行号（若可知）、问题描述和建议。

### P1 — 正确性

- 逻辑错误、条件判断错误、运算顺序错误
- 未处理的边界情况（空列表、零值、None/null、溢出）
- 竞态条件、并发不安全的共享状态
- API 使用错误（参数顺序、返回值误用、错误码未处理）
- 数据流错误（输入未校验就使用、输出未检查就信任）

### P2 — 安全性

- 注入风险（SQL 注入、命令注入、路径遍历）
- 不安全的数据存储（明文密码、敏感信息写入日志）
- 密钥/凭据硬编码或泄漏到版本控制
- 权限校验缺失或绕过

### P3 — 效率

- 循环内的重复计算（可提到循环外的常量计算）
- N+1 查询或重复 IO
- 不必要的深拷贝或大对象创建
- 内存泄漏（未释放的资源、循环引用）

### P4 — 可维护性

- 函数/方法过长，难以理解
- 命名不清晰（缩写、单字母变量用于非局部作用域）
- 重复代码（同一逻辑在多处重复，而非复用）
- 违反项目已有的代码风格或模式

### P5 — 测试

- 关键路径无测试覆盖
- 测试只覆盖正常路径，未覆盖错误路径和边界值
- 断言太弱（只断言不报错，未断言实际结果）

### P6 — 性能（负载下）

- 热路径上的低效操作（仅在确认是热路径时提出）
- 数据结构选择不当（线性扫描代替哈希查找等）

### P7 — 可扩展性

- 硬编码的上限或假设（假设数据量永远很小）
- 串行处理可并行化的场景

### P8 — 模块化与复用性

- 新代码重新实现了项目中已有的工具函数
- 新抽象与现有架构边界冲突

---

## 第四步：严重程度分级

对每个发现的问题，按以下标准打标：

| 级别 | 标记 | 含义 | 合并建议 |
|------|------|------|----------|
| 严重 | 🔴 | 生产故障、安全漏洞、数据损坏风险 | 必须修复后才能合并 |
| 高   | 🟠 | 明显 bug、显著性能退化、功能不符合预期 | 应修复 |
| 中   | 🟡 | 偏离最佳实践、技术债 | 建议改进 |
| 低   | 🟢 | 格式、拼写、注释等细节 | 作者自行决定 |

**分级规则（优先适用）：**

- 拼写错误 → 🟢
- 注释/文档字符串措辞 → 🟢
- 硬编码的非关键常量 → 🟢
- 测试文件中的问题 → 🟢 或 🟡（测试文件中的逻辑错误可升为 🟠）
- Markdown / 文档文件 → 🟢 或 🟡

---

## 评审原则

**只评审可验证的问题**：

- 每条评论必须基于可见代码中的具体事实，不写"请检查一下"或"可以考虑"。
- 不解释代码在做什么——只指出问题所在。
- 不重复相同模式：同一类问题只报告一次（可附注"其他类似位置同理"）。
- 每条评论包含：**问题说明**（是什么）+ **影响**（为什么有问题）+ **改进建议**（怎么修）。

---

## 第五步：输出审查报告

按以下格式输出结构化报告。报告完成后，由调用工具自行决定如何提交（`gh` CLI、MCP 工具、手动复制等）。

```
# PR 审查报告

**PR**：[PR 标题或链接]
**变更类型**：[feat / fix / refactor / ...]
**审查日期**：[日期]
**总体结论**：批准合并 / 条件合并（修复 🔴 后）/ 需要重大修改

---

## 总览

[2-3 句话：变更质量的整体评价，主要风险点]

---

## 问题清单

### 🔴 严重（必须修复）

- **文件**：`path/to/file.py`，第 N 行
  **问题**：[具体问题描述]
  **影响**：[为什么有问题]
  **建议**：[具体修改方式或代码片段]

[若无严重问题，写"无"]

---

### 🟠 高（应修复）

- **文件**：...
  **问题**：...
  **影响**：...
  **建议**：...

[若无，写"无"]

---

### 🟡 中（建议改进）

- ...

[若无，写"无"]

---

### 🟢 低（作者自行决定）

- ...

[若无，写"无"]

---

## 正面反馈

[列出值得肯定的实现点，1-3 条，避免泛泛而谈]

---

## 下一步

[给作者的明确行动项，按优先级排列]
```

---

## 特殊情况处理

**变更量很大（>500 行）**：先看 PR 描述和测试文件，了解意图后再看实现。重点审 P1/P2，其余选择性覆盖，在报告中说明审查范围。

**纯文档或配置变更**：跳过 P1–P3，重点检查准确性（链接是否有效、命令是否正确、配置值是否合理）。

**无法获取完整 diff**：在报告开头说明已审查的内容范围，以及因信息缺失而无法评审的部分。
